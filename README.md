CADI WEBSITE
Agentic Search — Setup & Configuration Guide
Digital Experience Team  |  April 2026
About this document: This document covers how to deploy the Claude-powered agentic search solution on any Drupal or static website. It includes GitHub Action setup, crawler configuration, the search interface, and maintenance guidance.

Contents



1. Overview
The CADI Agentic Search solution replaces the Drupal global site search with an AI-powered search agent. Rather than performing keyword matching, it uses a Large Language Model (Claude, by Anthropic) to understand user intent, match against an indexed snapshot of the website, and return ranked results with explanations.
The system operates in two parts: a scheduled crawl that snapshots the website into a JSON index stored on GitHub, and a search interface that queries that index using Claude's API.
1.1 Architecture
Component	Technology	Purpose
Site Crawler	Python (GitHub Action)	Crawls pages, extracts text/metadata, outputs search-index.json
Content Index	JSON on GitHub	Searchable snapshot of site content, refreshed weekly
Search Interface	HTML/React	User-facing search UI served on any domain
AI Search Agent	Claude API (Anthropic)	Understands queries, ranks results, explains relevance
Hosting	GitHub Pages or any CDN	Static file hosting for the search interface

1.2 Cost Considerations
Each search query calls the Anthropic API. As of April 2026, Claude Sonnet 4 pricing is approximately $0.003 per search (using ~1,000 output tokens + ~2,000 input tokens for a typical 300-page index). For 500 searches/day this equates to roughly £1.10/day or ~£400/year.
Cost / API Key: API keys are managed via Anthropic's Claude Platform. The search interface must be configured with a valid key. See Section 4 for key management guidance.

2. Prerequisites
2.1 Accounts & Access
•	GitHub account with a repository to store the crawler and index
•	Anthropic account — obtain an API key at console.anthropic.com
•	Access to the target website (no login required for public sites)
2.2 Repository Structure
Your GitHub repository should be structured as follows:
your-search-repo/
├── .github/
│   └── workflows/
│       └── crawl-site.yml       ← GitHub Action (provided)
├── scripts/
│   └── crawl.py                 ← Crawler script (provided)
├── search-index.json            ← Auto-generated by crawler
└── index.html                   ← Search interface (provided)


3. GitHub Action: Crawler Setup
3.1 Add the Workflow File
Copy crawl-site.yml into .github/workflows/ in your repository. This Action will run automatically every Sunday at 2am UTC, and can also be triggered manually from the GitHub Actions tab.
3.2 Configure the Target URL
The crawler defaults to the CADI website. To point it at a different domain, update the env section of the workflow file:
env:
  TARGET_URL: https://your-website.ac.uk
  MAX_PAGES: '300'
  STAY_ON_DOMAIN: 'true'

3.3 Key Crawler Parameters
Parameter	Default	Description
--url	(required)	Root URL to begin crawling
--max-pages	300	Maximum pages to index. Increase for large sites.
--stay-on-domain	true	Set false to follow external links (not recommended)
--output	search-index.json	Output file path

3.4 Customising Content Extraction for Drupal
The crawler uses CSS selectors to find main content areas, avoiding navigation and footers. For Drupal sites you may wish to add CADI-specific selectors to the CONTENT_SELECTORS list in crawl.py:
CONTENT_SELECTORS = [
    "main",
    ".main-content",
    ".field--name-body",      # ← Drupal standard content field
    ".node__content",         # ← Drupal node wrapper
    "[role='main']",
]

Drupal Tags Bug: If the Drupal tagging bug (tags not showing on content pages) is resolved, add .field--name-field-tags a to the tag extraction in the extract_meta() function to improve content type filtering.

4. Search Interface Setup
4.1 Deploying the Interface
The search interface is a single HTML file (index.html) that can be hosted anywhere — GitHub Pages, a CDN, or embedded directly into a Drupal page via an iFrame or block. No server-side code is required.
1.	Upload index.html to your chosen hosting location
2.	Set the GITHUB_RAW_INDEX_URL constant at the top of the file to point to your raw search-index.json on GitHub
3.	Set your Anthropic API key (see Section 4.2)
4.	Test by opening the page and performing a sample search

// In index.html, update these two constants:
const GITHUB_RAW_INDEX_URL =
  "https://raw.githubusercontent.com/your-org/your-repo/main/search-index.json";

const ANTHROPIC_API_KEY = "sk-ant-...";  // See Section 4.2

4.2 API Key Management
The API key can be configured in two ways depending on your security requirements:
Option A — Embedded (simplest): Set the key directly in index.html. Suitable for internal staff-only tools behind authentication. Not suitable for public-facing pages.
Option B — Environment variable via build step: If using a build pipeline (e.g. Vite, Next.js), inject the key at build time using environment variables. The key will not appear in source code.
Security Warning: Never commit an API key to a public GitHub repository. If using Option A, ensure the hosting location is access-controlled or the repository is private.

4.3 Embedding in Drupal
To replace or supplement the existing Drupal search, options include:
•	Create a new Drupal Basic Page and embed the interface using an HTML block or custom module
•	Add a Drupal menu link pointing to the hosted interface URL
•	Use Drupal's block system to add a compact search widget in the header
•	Redirect /search to the new interface using Drupal's redirect module

5. Features & Configuration
5.1 Priority Pages
Certain pages (e.g. the main CPD and Support page) can be boosted in search results by adding them to the priority_pages array in the system prompt configuration:
const PRIORITY_PAGES = [
  { url: "https://cadi.port.ac.uk/cpd-support",
    boost_terms: ["cpd", "professional development", "workshops", "training"] },
  { url: "https://cadi.port.ac.uk/fellowship",
    boost_terms: ["fellowship", "advancehe", "pgcert"] },
];

5.2 Content Gap Detection
The AI agent automatically reports a content_gap field when a query has no strong matches in the index. The interface surfaces these as admin notifications. To view gap reports, enable the admin panel in the interface settings — this logs all gaps to the browser console and optionally posts them to a webhook.
5.3 External Link Surfacing
Tools like APEX that exist as external links can be promoted by adding them to the external_links manifest in the index configuration:
const EXTERNAL_LINKS = [
  { name: "APEX", url: "https://apex.port.ac.uk",
    description: "Academic Practice Enhancement Platform",
    keywords: ["apex", "academic practice"] },
  { name: "Moodle", url: "https://moodle.port.ac.uk",
    keywords: ["moodle", "vle", "online learning"] },
];

5.4 Date Filtering
The search interface includes an optional date filter. This uses the date field extracted from page metadata during crawling. For Drupal sites, ensure your content type's published date is exposed in the page's <time datetime> tag or og:article:published_time meta tag.
5.5 Search Prompt Guidance
The interface includes configurable search prompt examples displayed below the input field. These guide users towards better queries:
const EXAMPLE_PROMPTS = [
  "CPD workshops this term",
  "How do I get AdvanceHE Fellowship?",
  "Assessment for large cohorts",
  "Panopto lecture recording setup",
];


6. Maintenance
6.1 Keeping the Index Fresh
The GitHub Action runs weekly by default. For sites with frequent content changes, adjust the cron schedule in crawl-site.yml. To trigger a manual re-index immediately, go to the GitHub repository > Actions tab > Crawl and Index Website > Run workflow.
6.2 Monitoring
•	Check GitHub Actions run history for crawl errors
•	Review the errors array in search-index.json for pages that failed to crawl
•	Monitor Anthropic API usage in the Anthropic console — set a monthly spend limit
6.3 Content Gap Backlog
Based on the GA data (Feb–Apr 2026), the following searches returned weak results due to missing content. These represent content improvement opportunities:

7. Reusing on Other Websites
The solution is domain-agnostic. To deploy on a new website, repeat these steps:
5.	Fork or copy the repository (or create a new one)
6.	Update TARGET_URL in crawl-site.yml to the new website's root URL
7.	Run the workflow manually to generate the initial search-index.json
8.	Update GITHUB_RAW_INDEX_URL in index.html to point to the new index file
9.	Review CONTENT_SELECTORS in crawl.py for any CMS-specific adjustments
10.	Optionally update PRIORITY_PAGES and EXTERNAL_LINKS in index.html
11.	Deploy index.html to the new site's hosting environment

Multi-site tip: One API key can serve multiple search interfaces. If deploying across many sites, monitor usage in the Anthropic console and consider per-site spend limits.
